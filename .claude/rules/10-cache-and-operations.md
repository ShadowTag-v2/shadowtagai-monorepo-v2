# Rule 10: Cache Architecture & Permission System

## The Cache Trick That Saves Real Money
The leak revealed how obsessively Anthropic treats prompt caching. Built-in tools are sorted as a contiguous prefix BEFORE MCP tools. So adding or removing MCPs doesn't blow your prompt cache. The system prompt is split at a static/dynamic boundary marker for the same reason.

### DANGEROUS_uncachedSystemPromptSection (Source-Verified)
`systemPromptSections.ts` defines two section types:
- **Cached sections** — computed once, stored in section cache, reused across turns.
- **Volatile sections** (`cacheBreak: true`) — recomputed EVERY turn, break the prompt cache when value changes.

The `DANGEROUS_` prefix is Anthropic's own internal warning marker. CLAUDE.md sits in a volatile section. This means:
- It is **re-read** every turn (not just session start)
- Editing it mid-session **breaks the prompt cache** for every subsequent message
- `resolveSystemPromptSections()` checks the cache first; volatile sections always bypass

### Write CLAUDE.md Once, Leave It Alone
**Operational rule:** Write your CLAUDE.md at session start and do not modify it during the session. Every edit triggers a full cache invalidation, wasting tokens and money. If you must update instructions mid-session, use the `/compact` command at a natural breakpoint immediately after.

If you're building on the API: tool ordering and prompt structure directly affect your bill. Treat cache invalidation as an accounting problem.

## The Hidden YOLO Mode Classifier
Beyond the 3-tier Allow/Deny/Prompt permission system, there's a YOLO classifier. It's a separate ML model that analyzes the conversation transcript and decides whether to auto-approve tool calls WITHOUT asking the user. It's gated behind a TRANSCRIPT_CLASSIFIER flag. Not shipped yet.

But the architecture is clear: the future of AI agents is one that never interrupts you unless it's genuinely unsure. Build your permission systems to be this adaptive, not just a static allowlist.
