# Workflow: Thread Recovery Protocol — Two Stage

**Trigger:** `/thread-recovery-2stage`

## Mission

Separate discovery from regeneration.

Do not proceed directly to rewriting.  
Audit first. Rebuild second.

---

## STAGE 1 — FORENSIC AUDIT AND RECOVERY ONLY

### Objective

Recover everything left on the table in the thread before changing any final output.

### Procedure

1. Re-read the full thread end to end.
2. Extract and enumerate:
   - every explicit request
   - every implicit requirement
   - every constraint
   - every dependency
   - every unresolved question
   - every abandoned branch
   - every partially completed deliverable
   - every place where code, analysis, or explanation was implied but not fully produced
3. Build a complete task ledger.
4. Search all available resources for missing context and evidence, including:
   - thread history
   - available files and artifacts
   - connected internal sources
   - Google Drive when available
   - relevant repositories
   - web research when useful
5. Identify all:
   - omissions
   - contradictions
   - changed assumptions
   - weak reasoning
   - missing code paths
   - overlooked implementation details
   - performance opportunities
   - maintainability improvements
   - accuracy improvements
   - leverage and monetization opportunities
6. Compare prior thread conclusions against recovered material.
7. Explain the meaningful distinctions:
   - asked vs answered
   - answered vs solved
   - assumed vs verified
   - explicit vs implied
   - mentioned vs operationalized
   - drafted vs implemented
   - local optimum vs global optimum
   - technically correct vs commercially useful
   - apparently complete vs actually complete

### Stage 1 Output Format

A. Recovery findings  
B. Complete task ledger  
C. Missing or incomplete items  
D. Newly recovered material  
E. Distinctions and changed assumptions  
F. What must be preserved, corrected, expanded, or replaced

### Critical Rule

- Do **not** regenerate answers yet.
- Do **not** regenerate code yet.
- Do **not** continue by inertia from the latest answer.
- Only audit, recover, compare, and reconcile.

### Stage-Gate Condition

Do not enter Stage 2 until:
- the task ledger is complete
- changed assumptions are explicitly reconciled
- missing and incomplete items are clearly enumerated

---

## STAGE 2 — REPLAN AND REGENERATE

### Objective

Rebuild the work from first principles using the reconciled truth from Stage 1.

### Procedure

1. Re-plan the solution cleanly using the full Stage 1 findings.
2. Correct broken assumptions.
3. Incorporate all recovered material.
4. Re-sequence the work for better clarity, implementation quality, and business outcome.
5. Regenerate the answers in improved form.
6. Reprint all relevant code in full, updated to match the revised plan.

### Stage 2 Output Format

A. Revised plan  
B. Regenerated answers  
C. Full updated code

### Critical Rule

- Prefer truth over continuity.
- Prefer elegance over thread momentum.
- Preserve prior wording only when it is still the best wording.

---

## Optimization Standard

Optimize the final result for:

- elegance
- simplicity
- correctness
- performance
- maintainability
- robustness
- accuracy
- user clarity
- business leverage
- financial upside
