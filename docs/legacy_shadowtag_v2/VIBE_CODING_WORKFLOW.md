# Vibe Coding Workflow Guide

**How a Senior Engineer at a $140M+ Startup Actually Codes with AI**

> *Adapted from real-world workflows at well-funded startups shipping 40% faster with 95% AI-assisted development*

## Overview

The "Vibe Coding" workflow represents a pragmatic approach to AI-assisted development that focuses on **system design and architecture** while leveraging AI for **repetitive implementation**. This isn't about letting AI make all decisions—it's about strategic delegation to maximize engineering impact.

### Core Principle

> **70% Rough Structure + Real-Time Refinement + AI-Reviewing-AI + Human-Controlled Testing = 40% Faster Delivery**

## The Four-Phase Workflow

### Phase 1: Rough Structure with Claude Code (Terminal)

**Goal**: Get 70% of the way there without getting stuck on implementation details

**Tools**: Claude Code in terminal
**Time**: 20-30% of development cycle
**Mindset**: Don't worry about perfection, just avoid getting blocked

#### How It Works

1. **Describe what you want in natural language**
   ```bash
   # Example: Starting Gemini Ingestion Layer feature
   $ claude

   > I need to build a multi-source crawler for the pnkln ingestion layer.
   > It should handle YouTube, Twitter, and News APIs concurrently.
   > Each source needs ethical crawling (robots.txt, rate limiting).
   > Output should be classified into Tier 1/2/3 based on relevance.
   > Target runtime: 45 minutes for nightly batch job.
   ```

2. **Let AI generate rough structure**
   - Scaffold main classes and modules
   - Define key interfaces and data models
   - Set up basic error handling
   - Create initial configuration structure

3. **Accept 70% solution**
   - Code compiles/runs but isn't polished
   - Architecture is sound but implementation rough
   - Edge cases may be missing
   - Documentation is sparse

### Phase 2: Real-Time Refinement with Cursor

**Goal**: Watch AI write code in real-time and catch hallucinations immediately

**Tools**: Cursor IDE
**Time**: 40-50% of development cycle
**Mindset**: Active supervision, not passive acceptance

#### How It Works

1. **Open rough structure in Cursor**
   - Load files generated in Phase 1
   - Identify areas needing refinement
   - Prioritize based on criticality

2. **Use Cursor's real-time code generation**
   ```
   # In Cursor, highlight the _crawl_source method and prompt:

   "Implement this method with:
   - Ethical crawling (check robots.txt)
   - Rate limiting (1 req/sec)
   - Exponential backoff for failures
   - Logging for debugging"
   ```

3. **Watch code appear line-by-line**
   - **Catch weird patterns immediately** (e.g., infinite loops)
   - **Verify logic as it's written** (e.g., off-by-one errors)
   - **Stop and redirect** if AI goes off track

4. **Iterate quickly**
   - Make small adjustments
   - Re-prompt for corrections
   - Build up polished implementation incrementally

#### Example Refinement (Phase 2)

```python
# src/ingestion/crawler_manager.py - Refined with Cursor

import asyncio
import logging
from typing import List, Dict
from datetime import datetime
import aiohttp
from urllib.robotparser import RobotFileParser

logger = logging.getLogger(__name__)

class CrawlerManager:
    """Manages multi-source crawling operations with ethical compliance."""

    def __init__(self, sources: List[str], rate_limit: float = 1.0):
        self.sources = sources
        self.rate_limit = rate_limit  # requests per second
        self.crawlers = self._initialize_crawlers()
        self._robot_parsers = {}

    async def run_nightly_batch(self) -> Dict:
        """Execute nightly crawl across all sources with concurrency."""
        start_time = datetime.now()
        logger.info(f"Starting nightly batch crawl for {len(self.sources)} sources")

        tasks = [
            self._crawl_source(source)
            for source in self.sources
        ]
        results = await asyncio.gather(*tasks, return_exceptions=True)

        aggregated = self._aggregate_results(results)
        runtime = (datetime.now() - start_time).total_seconds() / 60
        logger.info(f"Batch completed in {runtime:.2f} minutes")

        return aggregated

    async def _crawl_source(self, source: str) -> Dict:
        """Crawl single source with ethical compliance and error handling."""
        try:
            # Check robots.txt
            if not await self._can_crawl(source):
                logger.warning(f"Crawling disallowed by robots.txt: {source}")
                return {"source": source, "items": [], "error": "robots.txt blocked"}

            # Rate-limited crawling with exponential backoff
            items = []
            retry_count = 0
            max_retries = 3

            while retry_count < max_retries:
                try:
                    async with aiohttp.ClientSession() as session:
                        await asyncio.sleep(1 / self.rate_limit)  # Rate limiting
                        items = await self._fetch_items(session, source)
                        break
                except aiohttp.ClientError as e:
                    retry_count += 1
                    backoff = 2 ** retry_count
                    logger.warning(f"Retry {retry_count} for {source} after {backoff}s")
                    await asyncio.sleep(backoff)

            return {"source": source, "items": items, "count": len(items)}

        except Exception as e:
            logger.error(f"Failed to crawl {source}: {e}")
            return {"source": source, "items": [], "error": str(e)}

    async def _can_crawl(self, source: str) -> bool:
        """Check if crawling is allowed by robots.txt."""
        if source not in self._robot_parsers:
            rp = RobotFileParser()
            rp.set_url(f"{source}/robots.txt")
            try:
                rp.read()
                self._robot_parsers[source] = rp
            except:
                return True  # Allow if robots.txt unavailable

        return self._robot_parsers[source].can_fetch("*", source)
```

**Key Difference from Phase 1**: This code is production-ready with proper error handling, logging, rate limiting, and ethical compliance—not just scaffolding.

### Phase 3: AI-Reviewing-AI with CodeRabbit

**Goal**: Catch different types of issues than human review

**Tools**: CodeRabbit scode extension + GitHub app
**Time**: 10-15% of development cycle
**Mindset**: Let AI find the edge cases and anti-patterns you missed

#### How It Works

1. **Local quick checks with scode extension**
   ```bash
   # Run CodeRabbit scode extension in Cursor
   # Catches common issues before committing:
   # - Missing error handling
   # - Security vulnerabilities
   # - Performance anti-patterns
   # - Type inconsistencies
   ```

2. **Commit and push to feature branch**
   ```bash
   git add src/ingestion/crawler_manager.py
   git commit -m "Add ethical multi-source crawler manager

   - Implements robots.txt compliance
   - Rate limiting at 1 req/sec configurable
   - Exponential backoff for failures
   - Concurrent source processing"

   git push origin feature/multi-source-crawler
   ```

3. **CodeRabbit GitHub app analyzes PR**
   - Deeper analysis than local extension
   - Cross-file dependency checks
   - Architecture pattern validation
   - Security vulnerability scanning

4. **Review AI suggestions**
   - Accept valid improvements
   - Reject false positives (explain why)
   - Iterate on flagged issues

#### Example CodeRabbit Feedback

```markdown
## CodeRabbit PR Review

### Security Issues
⚠️ **High**: `RobotFileParser.read()` can execute arbitrary HTTP requests
   - Recommendation: Add timeout parameter (30s)
   - Add to line 87: `rp.read()` → `await asyncio.wait_for(rp.read(), timeout=30)`

### Performance
💡 **Medium**: Creating new `aiohttp.ClientSession` per source is inefficient
   - Recommendation: Share session across all sources in batch
   - Estimated improvement: 20% faster runtime

### Best Practices
✅ **Low**: Consider adding type hints for `_fetch_items` return value
   - Current: `async def _fetch_items(session, source)`
   - Suggested: `async def _fetch_items(session: ClientSession, source: str) -> List[Dict]`
```

**Why This Works**: AI-reviewing-AI catches different patterns:
- **Human review**: Architecture, business logic, maintainability
- **AI review (CodeRabbit)**: Security, performance, edge cases, type safety
- **AI generation (Claude/Cursor)**: Implementation speed, boilerplate reduction

This creates a **complementary review system** rather than redundant checking.

### Phase 4: Human-Controlled Testing

**Goal**: Comprehensive validation with human decision-making on production deployment

**Tools**: pytest, staging environment, manual QA
**Time**: 20-30% of development cycle
**Mindset**: AI helps write tests, humans decide when to ship

#### How It Works

1. **AI-assisted test generation**
   ```bash
   # In Cursor, prompt:
   "Generate comprehensive pytest tests for CrawlerManager:
   - Test robots.txt compliance (allowed/blocked scenarios)
   - Test rate limiting (verify 1 req/sec limit)
   - Test exponential backoff (simulate failures)
   - Test concurrent source processing
   - Mock external APIs"
   ```

2. **Human-written edge cases**
   ```python
   # tests/test_crawler_manager.py - Human adds edge cases AI missed

   def test_all_sources_fail_simultaneously():
       """What if every source is down? System should gracefully degrade."""
       # Human insight: AI generated happy path, missed total failure scenario
       pass

   def test_robots_txt_changes_mid_crawl():
       """What if robots.txt is updated during 45-min batch?"""
       # Human insight: AI assumed static robots.txt
       pass
   ```

3. **Run comprehensive test suite**
   ```bash
   pytest tests/ --cov=src/ingestion --cov-report=html
   # Target: >90% coverage (AI helps reach 70-80%, human adds 10-20%)
   ```

4. **Deploy to staging**
   ```bash
   # GKE staging cluster
   kubectl apply -f k8s/staging/ingestion-cronjob.yaml

   # Monitor first staging run
   kubectl logs -f cronjob/gemini-ingestion-staging
   ```

5. **Manual QA validation**
   - ✅ Verify ethical compliance (check logs for robots.txt adherence)
   - ✅ Confirm runtime within 45-min target
   - ✅ Validate tier classification quality (spot-check Tier 1/2/3 assignments)
   - ✅ Check cost tracking (should be ~$77/month estimate)

6. **Human decision: Ship or iterate**
   - **Ship**: If all metrics green and QA passes
   - **Iterate**: If issues found, return to Phase 2 for refinement

**Critical Point**: Deployment decisions stay with humans. AI helps validate, but engineers own the "go/no-go" call.

## Best Practices

### Do's ✅

1. **Use AI for scaffolding and boilerplate** (Phase 1)
2. **Supervise AI in real-time** (Phase 2)
3. **Let AI review AI code** (Phase 3)
4. **Keep deployment decisions human** (Phase 4)
5. **Focus engineering time on architecture and system design**
6. **Iterate quickly with AI, validate thoroughly with tests**

### Don'ts ❌

1. **Don't expect 100% perfect code from AI** (target 70% in Phase 1)
2. **Don't skip human review** (AI review complements, doesn't replace)
3. **Don't let AI make architectural decisions** (that's the engineer's job)
4. **Don't deploy without staging validation** (AI can't predict all real-world issues)
5. **Don't blindly accept AI suggestions** (verify logic and business requirements)

## Adapting This Workflow to Your Team

### For Small Teams (1-5 engineers)

- Start with **Phase 1 + 2** (Claude Code + Cursor)
- Add **Phase 3** (CodeRabbit) once code review becomes bottleneck
- Keep **Phase 4** (testing) human-driven from day one

### For Medium Teams (5-20 engineers)

- Adopt **all four phases** for consistency
- Standardize on same tools (avoid fragmentation)
- Create shared prompts for common patterns

### For Large Teams (20+ engineers)

- Enforce **Phase 3** (AI review) as gate before human review
- Build internal prompt libraries for domain-specific tasks
- Track metrics (speed, quality) to demonstrate ROI

## Common Questions

### Q: Doesn't AI-reviewing-AI seem redundant?

**A**: No, because different AI tools catch different issues:
- **Claude/Cursor** (generation): Fast implementation, general patterns
- **CodeRabbit** (review): Security, performance, cross-file dependencies
- **Human** (architecture): Business logic, system design, maintainability

They're complementary, not redundant.

### Q: How do junior engineers benefit?

**A**: By focusing on **design** instead of **implementation**:
- AI handles boilerplate and repetitive coding
- Junior engineers learn architecture and system thinking
- They deliver senior-level output by combining design skills with AI implementation
- Faster promotions because they demonstrate strategic thinking

### Q: What about AI hallucinations?

**A**: Caught in **Phase 2** (real-time refinement):
- Watch code appear line-by-line in Cursor
- Stop and redirect when AI goes off track
- Much easier than reviewing massive code dumps later

### Q: How do you handle sensitive code or trade secrets?

**A**: Use self-hosted AI or on-premise solutions:
- Deploy Claude Code with local models (if available)
- Use CodeRabbit's self-hosted enterprise version
- Keep critical IP in human-written code, use AI for infrastructure/boilerplate

### Q: Does this work for all types of coding?

**A**: Works best for:
- ✅ Web services and APIs (like pnkln stack)
- ✅ Data pipelines and ETL
- ✅ CRUD applications
- ✅ Infrastructure as code

Less effective for:
- ❌ Novel algorithms (AI lacks domain expertise)
- ❌ Performance-critical code (requires deep optimization)
- ❌ Safety-critical systems (human verification essential)

## Conclusion

The Vibe Coding workflow isn't about replacing engineers with AI—it's about **strategic delegation** to maximize engineering impact. By handling repetitive implementation with AI, engineers can focus on what humans do best: **system design, architecture, and strategic decision-making**.

**Key Takeaway**: 95% AI-assisted doesn't mean 95% autonomous. It means 95% of *implementation time* leverages AI, while 100% of *architectural decisions* remain human-driven.
