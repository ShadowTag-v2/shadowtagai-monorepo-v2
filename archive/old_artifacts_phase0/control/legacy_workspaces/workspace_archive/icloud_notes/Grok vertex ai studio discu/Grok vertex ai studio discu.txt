Overview of AutoGen
AutoGen is an open-source programming framework developed by Microsoft for building AI agents and enabling multi-agent collaboration to solve complex tasks. 0 1 It focuses on conversational patterns among agents, allowing them to reason, plan, and execute autonomously or with human input. 2 3 Originally released in 2023, it has evolved into versions like v0.2 and v0.4, with a rewrite emphasizing modularity and extensibility. 8 In the context of Google Cloud, AutoGen has been adapted as AG2 (formerly AutoGen) within Vertex AI Agent Engine, providing a high-level abstraction for LLM workflows. 31 15
Key Features
Multi-Agent Orchestration: Supports patterns like two-agent chats, group chats, and hierarchical teams where agents delegate tasks. 4 7 Agents can use tools, APIs, or custom functions for real-world actions. 5
Model Flexibility: Works with various LLMs, including OpenAI, Azure OpenAI, and Google’s Gemini via Vertex AI. 9 32 It’s model-agnostic but optimized for conversational workflows.
Tool Integration: Includes built-in support for code execution, web browsing (via MCP/Playwright), and extensions for third-party tools. 20 12
Human-in-the-Loop: Allows seamless integration of human feedback during agent interactions. 6
Extensibility: Core API for low-level control, AgentChat API for prototyping, and extensions for custom capabilities. 20
AutoGen is particularly strong for agentic AI in areas like automation, research, and problem-solving, outperforming single-agent setups in benchmarks. 11
Compatibility with Vertex AI Workbench
AutoGen is fully compatible with Vertex AI Workbench, Google’s managed Jupyter notebook environment, as it’s Python-based and can be installed via pip. 9 32 Workbench supports Python 3.10+, which aligns with AutoGen’s requirements. 20 Key integrations include using Gemini models via Vertex AI APIs, with enhanced authentication for enterprise use. 9 32 As AG2 in Vertex AI Agent Engine, it offers templates for rapid agent development. 31 15 Limitations: No native pre-installation in Workbench, so manual setup is needed; relies on Google Cloud authentication. 32 Examples like weather agents have been demonstrated in Colab, which translates directly to Workbench notebooks. 29
Compared to other frameworks, AutoGen/AG2 excels in multi-agent flexibility within Vertex AI, though it may require more setup than no-code options like Vertex AI Agent Builder. 10 11 18

  import time
  from google.api_core.exceptions import ResourceExhausted

  def safe_invoke(llm, messages, retries=3):
      for attempt in range(retries):
          try:
              return llm.invoke(messages)
          except ResourceExhausted:
              time.sleep(60)
  ```

“Discussion on Code Execution Tool Integration in Gemini Configurations
The phrase “For advanced features like code execution tools, integrate Tool(code_execution=ToolCodeExecution()) in configs” refers to enabling Gemini’s native code execution capability, a built-in tool that allows the model to autonomously generate, execute, and iterate on Python code during response generation. 23 This is particularly useful for tasks involving computation, data analysis, or verification (e.g., summing primes, solving integrals), where the model can self-correct based on execution results without external intervention. 0 1 4 Unlike traditional agent frameworks where code is generated and run externally (e.g., by a separate executor agent), this integrates execution directly into the model’s reasoning loop, reducing latency and hallucinations in math-heavy or logic-based queries. 5 6
How It Works in the Gemini API
Core Mechanism: The model interleaves reasoning with code: it plans code, runs it in a sandboxed environment (with access to libraries like NumPy, SciPy, Matplotlib), and incorporates outputs back into its response. 23 This is iterative—e.g., if initial code fails, the model refines it.
Integration Syntax: In Python (via the google-genai SDK), you define the tool as follows: from google.genai import types

tools = [types.Tool(code_execution=types.ToolCodeExecution())]
 Then pass it to the generation config: response = client.models.generate_content(
    model="gemini-3.1-flash",
    contents="Sum the first 50 primes.",
    config=types.GenerateContentConfig(tools=tools)
)
 The response includes parts like executable_code (the generated script) and code_execution_result (output or errors). 23 Similar patterns exist for streaming, chat sessions, and other languages (JS, Go, cURL). 23
Vertex AI Variant: For cloud-based setups, use: from vertexai.generative_models import Tool, CodeExecution

tool = Tool(code_execution=CodeExecution())
model = GenerativeModel("gemini-3.1-flash", tools=[tool])
 This leverages Vertex AI’s enhanced security and scaling. 1 4
Integration with Frameworks like AutoGen
In AutoGen, which builds multi-agent systems around LLMs, code execution is typically handled externally via UserProxyAgent‘s code_execution_config—the assistant (Gemini) generates code, and the proxy runs it locally. 24 25 However, to leverage Gemini’s internal code execution (for more seamless, model-driven iteration), you integrate the tool into the LLM config:
Config Adjustment: Add the tool to the config_list or llm_config by extending the Gemini client wrapper. For Vertex AI: config_list = [
    {
        "model": "gemini-1.5-pro",
        "api_type": "google",
        "project_id": "your-project",
        "location": "us-west1",
        "tools": [Tool(code_execution=CodeExecution())]  # Integrate here
    }
]
assistant = AssistantAgent("assistant", llm_config={"config_list": config_list})
 This passes the tool to Gemini’s generate_content calls under the hood. 8 12 For AI Studio (non-Vertex), use the OpenAI-compatible endpoint and include tools in the request kwargs. 9
Benefits in AutoGen: Enables agents to handle complex tasks (e.g., math puzzles, data viz) with fewer rounds, as Gemini self-executes. 2 3 Challenges include limited multi-tool support in early versions (e.g., single-turn tool calls) and potential sandbox restrictions. 7
Example Use Case: In a multi-agent setup, a Gemini assistant with code execution could compute integrals autonomously before passing results to other agents for visualization. 25
This integration shifts from agent-orchestrated execution to model-native, making it “advanced” for production workflows where reliability and efficiency matter.

Define Tools (e.g., weather forecast): from langchain_core.tools import tool
from geopy.geocoders import Nominatim
from pydantic import BaseModel, Field
import requests

geolocator = Nominatim(user_agent="weather-app")

class SearchInput(BaseModel):
    location: str = Field(description="The city and state, e.g., San Francisco")
    date: str = Field(description="The forecasting date in format (yyyy-mm-dd)")

@tool("get_weather_forecast", args_schema=SearchInput, return_direct=True)
def get_weather_forecast(location: str, date: str):
    location = geolocator.geocode(location)
    if location:
        try:
            response = requests.get(
                f"https://api.open-meteo.com/v1/forecast?latitude={location.latitude}&longitude={location.longitude}&hourly=temperature_2m&start_date={date}&end_date={date}"
            )
            data = response.json()
            return {time: temp for time, temp in zip(data["hourly"]["time"], data["hourly"]["temperature_2m"])}
        except Exception as e:
            return {"error": str(e)}
    else:
        return {"error": "Location not found"}

tools = [get_weather_forecast]

Initialize Gemini and Bind Tools: from langchain_google_genai import ChatGoogleGenerativeAI
from datetime import datetime

llm = ChatGoogleGenerativeAI(
    model="gemini-3.1-flash",
    temperature=1.0,
    max_retries=2
)

Define Nodes and Edges: from langchain_core.messages import ToolMessage
from langchain_core.runnables import RunnableConfig

tools_by_name = {tool.name: tool for tool in tools}

def call_tool(state: AgentState):
    outputs = []
    for tool_call in state["messages"][-1].tool_calls:
        tool_result = tools_by_name[tool_call["name"]].invoke(tool_call["args"])
        outputs.append(
            ToolMessage(
                content=tool_result,
                name=tool_call["name"],
                tool_call_id=tool_call["id"],
            )
        )
    return {"messages": outputs}

def call_model(state: AgentState, config: RunnableConfig):
    response = model.invoke(state["messages"], config)
    return {"messages": [response]}

def should_continue(state: AgentState):
    messages = state["messages"]
    if not messages[-1].tool_calls:
        return "end"
    return "continue"

Build and Run the Graph: from langgraph.graph import StateGraph, END

workflow = StateGraph(AgentState)
workflow.add_node("llm", call_model)
workflow.add_node("tools", call_tool)
workflow.set_entry_point("llm")
workflow.add_conditional_edges(
    "llm",
    should_continue,
    {"continue": "tools", "end": END},
)
workflow.add_edge("tools", "llm")
graph = workflow.compile()

inputs = {"messages": [("user", f"What is the weather in Berlin on {datetime.today()}?")]}
for state in graph.stream(inputs, stream_mode="values"):
    last_message = state["messages"][-1]
    last_message.pretty_print()

This creates a cyclic workflow where Gemini reasons, calls tools if needed, and iterates until resolved. 0
