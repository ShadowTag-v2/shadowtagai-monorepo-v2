🧩 PLAN MODE STYLE GUIDE v1.0

Purpose:
To generate hyper-concise, auto-parsable, low-friction plans that maximize semantic density and minimize context drift.
This guide formalizes how your planning steps should look before execution.

⸻

🧠 Core Philosophy

“Sacrifice grammar for concision. Surface decisions last.”

	1.	Concision > Grammar: Every token must carry an action, state, or decision.
	2.	Line = atomic instruction: One line, one outcome. No conjunctions.
	3.	All verbs imperative: e.g., “Add param,” “Refactor func,” “Defer X to Phase 2.”
	4.	Readable at a glance: You should grasp the entire plan in <5 seconds.
	5.	End with open questions or blockers. They’re always surfaced, never buried.

⸻

📐 Structure Template

<package or module>:
- <action> → <target>
- <next action> → <target>

Impl:
- <approach summary>
- <temporary compat plan>
- <phase boundaries>

Unresolved Qs:
- <open decision 1>?
- <open decision 2>?

Options:
1. Proceed + auto-apply edits
2. Proceed + manual approve
3. Hold + continue plan


⸻

✍️ Writing Rules

Rule	Example	Notes
Drop filler words	“Change handler → async wrapper”	No “the,” “to,” or “will.”
Prefer → over = or →	“opts.db → opts.adapter”	Symbolic clarity for direction of change.
No full stops	“update server params → include adapter”	Avoid sentence closure — implies ongoing process.
Capitalize only proper nouns / code refs	“EvaliteAdapter ok; SQLite ok”	Reduces noise, aids scanning.
Keep 60 char line width max	“update evalite opts → adapter param”	Cursor + Claude render nicely in narrow panes.


⸻

🧩 Syntax Keys

Symbol	Meaning
→	Transformation / migration
=	Assignment or decision
+	Add / append / include
-	Remove / deprecate
?	Question or unresolved
//	Optional inline note
⚠️	Risk or dependency warning
Phase 1/2	Temporal grouping for rollout

🧮 Example — Applied

packages/evalite/src/server.ts:
- change createServer(opts:{db:SQLiteDB}) → {adapter:EvaliteAdapter}
- update all opts.db refs → opts.adapter
- inline adapter wrapper for backward compat
- log adapter usage w/ debug flag
- full SQLite adapter extraction = Phase 2

Impl:
- create adapter iface matching DB ops
- integrate wrapper in runEvalite()
- confirm parity w/ legacy path
- phase 2: move adapter into separate module

Unresolved Qs:
- export-static.ts adopt adapter now or defer?
- inline logging stay local or move to reporter?
