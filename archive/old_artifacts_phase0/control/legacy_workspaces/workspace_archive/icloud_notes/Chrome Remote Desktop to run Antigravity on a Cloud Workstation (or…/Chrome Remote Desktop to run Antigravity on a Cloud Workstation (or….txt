Why would I want to run Antigravity remotely?

This is probably the first question you would ask yourself. Of course, many of us could just download the binaries and run Antigravity directly on their developer workstation. A couple of reasons why you might want to consider running it in a remote environment include:

    Corporate policies or hurdles: If you are not allowed to install software or have to configure three levels of forward proxies to reach the internet this could potentially be an option to get your hands on this shiny new toy.
    Resource Requirements: Running multiple agents in parallel in a strong suit of Antigravity but can also be taxing on your system resources, especially if you are not issued with a fully-specced, top-of-the-line machine or opted for something light and portable to still be able to work in a crammed economy class seat. A remote machine can deliver the resources you need in a very economical pay as you go model.
    Sandboxing: Especially in an agentic world the ability to sandbox is not to underestimate. Running your agentic helpers in a remote machine provides a very high-degree of sandboxing.

Customizing the Workstation Image

Because I want to run my Antigravity on a Cloud Workstation I start with the Cloud Workstation base image and add customization layers on top.

The Dockerfile contains all the steps to:

    Install the dependencies and a lightweight desktop environment.
    Install Chrome Remote Desktop (the local agent for connecting remotely) and Chrome itself as it will be used by Antigravity.
    (Hackish I know) Mock sysctl to make sure the sysctl commands return successfully even in a containerized environment.
    Install Antigravity and pre-configure some parameters to make it easier to launch later. If you wanted to run another UI-based application like Android Studio, you could replace or augment these instructions here.
    Copy the runtime configuration script that will initialize the config in the user’s persistent home directory.

FROM us-central1-docker.pkg.dev/cloud-workstations-images/predefined/base

ARG DEBIAN_FRONTEND=noninteractive

# 1. Install System Dependencies and XFCE
RUN apt-get update && apt-get install -y \
    xvfb \
    xfce4 \
    xfce4-goodies \
    xbase-clients \
    dbus-x11 \
    psmisc \
    python3-psutil \
    xserver-xorg-video-dummy \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

# 2. Install Google Chrome and Chrome Remote Desktop
RUN wget -q -O - https://dl.google.com/linux/linux_signing_key.pub | gpg --dearmor -o /usr/share/keyrings/google-chrome-keyring.gpg && \
    echo "deb [arch=amd64 signed-by=/usr/share/keyrings/google-chrome-keyring.gpg] http://dl.google.com/linux/chrome/deb/ stable main" | tee /etc/apt/sources.list.d/google-chrome.list > /dev/null && \
    echo "deb [arch=amd64 signed-by=/usr/share/keyrings/google-chrome-keyring.gpg] https://dl.google.com/linux/chrome-remote-desktop/deb stable main" | tee /etc/apt/sources.list.d/chrome-remote-desktop.list > /dev/null && \
    apt-get update && \
    apt-get install -y google-chrome-stable chrome-remote-desktop && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# Chrome Sandbox Fix
# Chrome will crash in Docker without --no-sandbox. 
# We divert the binary to force this flag.
RUN dpkg-divert --add --rename --divert /usr/bin/google-chrome-stable.real /usr/bin/google-chrome-stable && \
    echo '#!/bin/bash' > /usr/bin/google-chrome-stable && \
    echo 'exec /usr/bin/google-chrome-stable.real --no-sandbox --disable-dev-shm-usage "$@"' >> /usr/bin/google-chrome-stable && \
    chmod +x /usr/bin/google-chrome-stable

# 3. Mock systemctl
RUN echo '#!/bin/bash' > /usr/bin/systemctl \
    && echo 'echo "Mock systemctl: $@"' >> /usr/bin/systemctl \
    && echo 'exit 0' >> /usr/bin/systemctl \
    && chmod +x /usr/bin/systemctl

# 4. Install Antigravity
RUN mkdir -p /etc/apt/keyrings && \
    curl -fsSL https://us-central1-apt.pkg.dev/doc/repo-signing-key.gpg | gpg --dearmor -o /etc/apt/keyrings/antigravity-repo-key.gpg && \
    echo "deb [signed-by=/etc/apt/keyrings/antigravity-repo-key.gpg] https://us-central1-apt.pkg.dev/projects/antigravity-auto-updater-dev/ antigravity-debian main" | tee /etc/apt/sources.list.d/antigravity.list > /dev/null && \
    apt-get update && \
    apt-get install -y antigravity && \
    dpkg-divert --add --rename --divert /usr/bin/antigravity.real /usr/bin/antigravity && \
    echo '#!/bin/bash' > /usr/bin/antigravity && \
    echo 'exec /usr/bin/antigravity.real --no-sandbox --disable-dev-shm-usage --disable-gpu "$@"' >> /usr/bin/antigravity && \    
    chmod +x /usr/bin/antigravity

# 5. Runtime customization scripts
COPY startup-scripts/ /etc/workstation-startup.d/
RUN chmod +x /etc/workstation-startup.d/*

The script that we keep in the startup-scripts folder next to the Dockerfile has the following content to configure the permissions and ensure the remote desktop agent is launched on subsequent workstation launches and after the initial authorization has been completed.

#!/bin/bash
set -e

# --- 1. Variables ---
TARGET_USER="user"
HOME_DIR="/home/$TARGET_USER"
CONFIG_DIR="$HOME_DIR/.config/chrome-remote-desktop"

# --- 2. Fix Permissions & Directories (Runtime) ---
mkdir -p "$HOME_DIR/.pki/nssdb"
chown -R $TARGET_USER:$TARGET_USER "$HOME_DIR/.pki"
chmod 700 "$HOME_DIR/.pki"

groupadd -f chrome-remote-desktop
usermod -aG chrome-remote-desktop $TARGET_USER

# --- 3. Setup Session File ---
if [ ! -f "$HOME_DIR/.chrome-remote-desktop-session" ]; then
    echo "Creating default .chrome-remote-desktop-session file."
    cat <<'EOF' > "$HOME_DIR/.chrome-remote-desktop-session"
#!/bin/bash
exec > /tmp/chrome-session.log 2>&1

# 1. Clean up environment from previous failed attempts
unset DBUS_SESSION_BUS_ADDRESS
unset XDG_RUNTIME_DIR

# 2. Set XFCE variables
export DESKTOP_SESSION=xfce
export GDMSESSION=xfce
export XDG_CURRENT_DESKTOP=XFCE
# Vital for finding default configs in a container
export XDG_CONFIG_DIRS=/etc/xdg:/etc/xdg/xfce4

# 3. Clean up stale session cache (prevents "failsafe" loops)
rm -rf ~/.cache/sessions

# 4. Start XFCE wrapped in its own DBus session
# This ensures xfconfd can always talk to the bus
echo "Starting XFCE with dbus-launch..."
exec /usr/bin/dbus-launch --exit-with-session /usr/bin/startxfce4
EOF
    chown $TARGET_USER:$TARGET_USER "$HOME_DIR/.chrome-remote-desktop-session"
    chmod +x "$HOME_DIR/.chrome-remote-desktop-session"
fi

# --- 4. Setup DBus (Runtime) ---
if [ ! -f /var/lib/dbus/machine-id ]; then
    mkdir -p /var/lib/dbus
    dbus-uuidgen > /var/lib/dbus/machine-id
fi

mkdir -p /var/run/dbus
rm -f /var/run/dbus/pid
dbus-daemon --system --fork

mkdir -p "$CONFIG_DIR"
chown -R $TARGET_USER:$TARGET_USER "$HOME_DIR"

# --- 5. Start Service ---
if ls "$CONFIG_DIR"/host*.json >/dev/null 2>&1; then
    echo "Starting Chrome Remote Desktop Service..."
    sudo -H -u $TARGET_USER bash -c "/opt/google/chrome-remote-desktop/chrome-remote-desktop --start" &
else
    echo "No configuration file matching $CONFIG_DIR/host*.json found. Skipping service start."
    echo "SSH into this container after launch and follow the instructions at https://remotedesktop.google.com/headless to register the remote service for the first time."
fi

Building the Cloud Workstation Image and Configuration:

Building the workstation image with Cloud Build and pushing it to Artifact Registry is as simple as running the following command:

gcloud builds submit --region=europe-west1 \
--tag europe-west1-docker.pkg.dev/$GCP_PROJECT_ID/infra/antigravity:latest \
--project $GCP_PROJECT_ID

You can of course also use other build methods and manually push it to the Artifact Registry if you prefer that approach.

Once the image is in Artifact Registry, you can turn to the Cloud Workstation setup. If you don’t have a Cloud Workstation cluster, run the following command to create one. If you already have a cluster you can skip it and use the existing cluster to save on the cluster fee.

gcloud workstations clusters create workstation-cluster \
--region=$REGION \
--network="projects/$GCP_PROJECT_ID/global/networks/default" \
--subnetwork="projects/$GCP_PROJECT_ID/regions/$REGION/subnetworks/default" \
--project=$GCP_PROJECT_ID

We then need to create a workstation configuration and service account in this cluster that uses our custom image:

gcloud iam service-accounts create workstation-vm-sa \
    --project=$GCP_PROJECT_ID

gcloud artifacts repositories add-iam-policy-binding dev-tooling \
    --project=$GCP_PROJECT_ID \
    --location=$REGION \
    --member="serviceAccount:workstation-vm-sa@$GCP_PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/artifactregistry.reader"

gcloud workstations configs create antigravity-config \
--cluster=workstation-cluster \
--region=$REGION \
--machine-type=e2-standard-8 \
--boot-disk-size=200 \
--container-custom-image=europe-west1-docker.pkg.dev/$GCP_PROJECT_ID/infra/antigravity:latest \
--service-account=workstation-vm-sa@$GCP_PROJECT_ID.iam.gserviceaccount.com \
--shielded-integrity-monitoring \
--shielded-secure-boot \
--shielded-vtpm \
--project=$GCP_PROJECT_ID

Create and initialize your Cloud Workstation

With the configuration in place you can either use the UI or the following gcloud command to create a brand new cloud workstation:

gcloud workstations create my-antigravity-ws \
--cluster=workstation-cluster \
--config=antigravity-config \
--region=$REGION \
--project=$GCP_PROJECT_ID

In the workstations section of the Google Cloud Console you should now see the workstation you just created:

Chrome Remote Desktop Connection:

The Remote Desktop Connection part is quite simple. You need to head to the Remote Desktop headless setup page and click “Begin” on the setup dialog.

Begin setting up a Chrome Remote Desktop

Ignore the instructions to install Chrome Remote Desktop on your remote computer (the container build took care of that for you) and click “Next” and then “Authorize” to get to the authorization commands. Note that these commands contain a sensitive — code argument that represents a temporary token that ties it to your account. You’ll want to copy the entire snippet under “Debian Linux” and paste it to your Cloud Workstation SSH terminal from before.

When you copy it to your workstation terminal, consider adding a memorable display name with --display-name=my-antigravity-ws (you’ll thank me later.)

user@my-antigravity-ws:~$ DISPLAY= /opt/google/chrome-remote-desktop/start-host \
--code="XXX" \
--redirect-url="https://remotedesktop.google.com/_/oauthredirect" \
--name=$(hostname) \
--display-name=my-antigravity-ws

The dialog will ask you for a 6-digit pin. Make sure you’ll remember this because it will be used later when you log-in with your Cloud Workstation. When everything is done you should see a message that confirms the host is ready.

(Optional) Running the Antigravity Image Locally

For me, the option of running an Antigravity Image with UI locally was more of a proof of concept and quick smoke test during the container development. But it did prove that technically you could also run the image that we built (or a slimmer version of it without the Cloud Workstation dependencies) locally to help with the sandbox concern that we mentioned at the beginning of this post. Obviously the enterprise and resource constraints still apply in this case.

The steps to run this are quite simple:

# Build the image
docker build -t antigravity-local

# Run the container interactively with a mounted /home directory
docker run -it --rm -v "$(pwd)/chrome-data:/home/user:rw" antigravity-local

# Get an interactive shell to run the Chrome Remote Desktop initialization
docker exec -it <container id> bash

Summary and next Steps

By moving your Antigravity environment to Cloud Workstations, you’ve created a powerful, portable, and secure “agentic sandbox.” You no longer have to worry about local resource contention or corporate proxy hurdles. You have a high-performance desktop accessible from any browser tab via Chrome Remote Desktop.
What’s next?

Now that you have a stable remote environment, here are three ways to take it further:

    Explore MCP Servers: Antigravity shines when it can interact with your tools. Try connecting it to MCP Toolbox for Databases to let your agents reason over your actual data in a secure cloud perimeter.
    Implement a continuous Cloud Build pipeline to keep your workstation image fresh and secure
    Automate with Terraform: If you plan on rolling this out to a larger team, consider Terraforming your Cloud Workstation configuration.
